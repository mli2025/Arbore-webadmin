# Arbore Web Admin 使用说明（Portainer 架构版）

> 本文面向**最终使用者与培训讲师**，说明在引入 **Portainer CE** 作为 Docker 引擎统一入口后，Web Admin 的用法与注意事项。可据此拆成 PPT 章节逐页讲解。

---

## 1. 产品定位（建议 PPT：封面 + 一页「是什么」）

- **Arbore Web Admin**：浏览器内的「AI Host / 本地服务」管理控制台。
- **能做什么**：查看标准服务状态、一键打开各服务 Web 入口、管理许可证、**上传镜像 tar 部署自定义服务**、查看资源与日志等。
- **不能做什么（重要）**：不在界面里改宿主机的 `docker-compose.yml` 或 Nginx 配置来部署自定义服务；自定义服务的生命周期由后端通过 **Portainer API** 与 Docker 引擎交互完成。

---

## 2. 为什么要接 Portainer（建议 PPT：一页「架构价值」）

| 以前（旧方式） | 现在（Portainer 方式） |
|----------------|------------------------|
| 后端直接碰 Docker Socket / 大文件一次性读入内存 | 大镜像 **流式** 交给 Portainer 的 `/images/load`，降低内存与超时风险 |
| 改 compose、改 Nginx 易不一致 | 自定义容器由 API 创建，**标签（Label）** 标识归属，行为可预期 |
| 失败原因难查 | 上传过程 **分阶段进度** + 日志与诊断接口，便于定位 |

**使用者需要知道的结论**：只要 **Portainer 可用、API Key 正确、环境 ID 正确**，自定义服务部署应比旧方案更稳定；若失败，界面会提示阶段，管理员可结合日志排查。

---

## 3. 访问与整体布局（建议 PPT：一页「界面导览」）

1. 浏览器打开部署地址（常见为 `http://<服务器IP>:9000`，具体以现场为准）。
2. 顶部：品牌区 + **当前版本号**（可点击查看版本说明）。
3. 主区域为 **标签页**：
   - **服务状态**：标准服务卡片（状态、健康、端口、访问、启停等）。
   - **资源监控**：主机资源概览。
   - **日志查看**：集中看日志（按现场配置）。
   - **系统配置**：系统侧参数（按现场配置）。
   - **许可证**：注册与授权；未授权时部分操作会禁用。
   - **自定义服务**：上传 tar、配置端口/环境变量/卷等，部署自有容器服务。

---

## 4. 标准服务（「服务状态」页）（建议 PPT：1～2 页）

### 4.1 卡片上能看到什么

- **显示名、描述、图标**：便于识别业务含义（非 Docker 原始容器名的直译）。
- **容器名（key）**：与真实容器名称对应，用于和后台状态对齐。
- **状态 / 健康 / 端口映射**：来自引擎侧查询；无 Web 入口的服务可将端口配置为 `0`，此时「访问服务」不可用属正常。

### 4.2 右上角齿轮：标准服务「展示配置」

- 点击 **齿轮** 打开配置表，可 **增加 / 编辑 / 删除** 标准服务条目。
- 每条包含：**容器名、显示名、描述、图标、宿主机端口**（用于「访问服务」打开的 URL：`当前协议 + 当前主机名 + 端口`）。
- 配置保存在服务器项目目录下的 **`config/standard-services.json`**（由后端读写）；**不在此界面创建或删除真实容器**，只改「看哪些、怎么显示、点哪个端口打开」。
- 若配置里有一条容器名，但环境里 **没有对应容器**：仍会显示卡片，状态为 **未找到**，便于提前占位或发现部署遗漏。

### 4.3 启停与许可证

- 部分服务支持 **启动 / 停止 / 重启**（与类型有关，例如 systemd 或以约定前缀命名的服务）。
- **许可证无效**时，标准服务的启停类操作可能被禁用，需先到「许可证」完成注册。

## 5. 自定义服务（核心）（建议 PPT：2～4 页）

### 5.1 使用场景

- 已有 **Docker 镜像 tar**（`docker save` 产物），希望在内网一键加载并跑起来，并映射到指定 **宿主机端口**。

### 5.2 推荐操作步骤（培训时可按步骤演示）

1. 打开 **自定义服务**，确认许可证有效（否则无法部署）。
2. 点击 **添加服务**（或等价入口）。
3. 填写 **服务名称**（小写字母、数字、连字符，规则以界面为准）。
4. 填写 **宿主机端口**、**容器内端口**（若与宿主机端口不同须填写容器端口）。
5. 按需配置 **环境变量、卷挂载、内存限制、说明文档 PDF**。
6. 选择 **镜像 tar 文件**，点击部署。
7. 观察 **部署进度** 弹窗：
   - **上半部分**：浏览器 → Web Admin 的上传字节进度（百分比、已传/总大小、速度）。
   - **下半部分**：服务器侧阶段（校验 → 传到 Portainer 并加载镜像 → 创建容器 → 启动容器 → 可选保存文档）。
8. 成功后列表刷新；可通过端口或现场约定的方式访问服务。

### 5.3 镜像与命名（讲师必讲，减少失败率）

- Docker 对镜像引用格式有严格要求；若 tar 内镜像名 **不符合规范**（常见：**仓库名含大写字母**、奇怪符号等），后台会自动将加载结果 **规范重标记** 为形如 **`arbore-<服务名>:latest`** 再创建容器，避免「invalid reference format」类错误。
- 建议团队规范：`docker save` 时使用 **全小写** 仓库名与清晰 tag，例如 `myteam/myapp:1.0.0`。

### 5.4 失败时怎么看（建议 PPT：一页「排障思路」）

- 进度弹窗会在 **具体阶段** 显示错误文案，不要只看浏览器笼统的 502。
- 管理员在服务器查看 **web-admin-api** 容器日志，可看到 Portainer 调用与镜像加载相关记录。
- 版本接口旁可配合运维使用诊断接口（由部署方提供具体 URL）：用于确认 **Portainer 是否可达、API Key、Endpoint ID** 等（以实际部署文档为准）。

## 6. 与 Portainer 相关的部署要点（给讲师的「幕后一页」，可选）

以下内容给 **运维/讲师**，不必对普通使用者逐条展开，但 PPT 可留一页「环境前提」：

- Web Admin **后端**通过环境变量连接 Portainer，典型包括：
  - `PORTAINER_URL`：例如 compose 网络内 `http://portainer:9000` 或实际服务名。
  - `PORTAINER_API_KEY`：在 Portainer 用户设置中创建 API Key。
  - `PORTAINER_ENDPOINT_ID`：Portainer 中对应本机 Docker 环境的数字 ID（常见为 `1`，以现场为准）。
- 若曾将 Portainer 容器 **改名**（例如品牌化后服务名变化），必须同步修改 `PORTAINER_URL` 中的主机名并重启 **web-admin-api**，否则自定义服务与状态查询会失败。
- 前端 Nginx 对 `/admin-api` 通常配置了 **大文件上传与较长超时**；超大 tar 仍需预留磁盘与耐心。

## 10. 发版必检清单（Release Checklist）

> **每一次合并入主分支或对外发版，都必须保证下面三处版本号一致，并且 changelog 已更新。**

### 10.1 三处版本号同步

| 位置 | 含义 | 备注 |
|------|------|------|
| `backend/main.py::API_VERSION` | 运行时返回 `/api/v1/version` 的版本 | 权威源 |
| `version.json` | OTA 客户端拉取的版本清单 | 必须与 `API_VERSION` 一致 |
| `frontend/src/App.vue::apiVersion` + `changelog` 列表 | 顶部版本徽章 + 「changelog」弹窗 | 必须包含本版本号的条目 |

### 10.2 一条命令同步（不要再手改 BUILD_TIME / version.json）

```powershell
# 步骤 1：手动改 backend/main.py 的 API_VERSION（语义化升版）
# 步骤 2：手动在 frontend/src/App.vue 的 changelog 数组顶部加一条新版本说明
# 步骤 3：运行同步脚本
py update-build-time.py
```

脚本会自动：

- 把 `BUILD_TIME` 刷成当前 UTC ISO-8601 时间戳
- 把 `version.json` 的 `version` 与 `build_time` 同步到与 `API_VERSION` 一致
- 校验 `App.vue` 是否已经包含本版本号的 changelog 条目，**漏写会输出警告**

### 10.3 校验模式（建议接到 CI / pre-commit）

```powershell
py update-build-time.py --check
```

任何不一致退出码非 0：

- 退出码 `2`：`App.vue` 找不到当前版本号或 changelog 缺条目
- 退出码 `3`：`version.json` 与 `API_VERSION` 不一致

### 10.4 changelog 写法约定

`frontend/src/App.vue` 的 `changelog` 数组顶部加一项即可，格式：

```js
{ version: 'v1.2.1', date: '2026-05-01', title: 'one-line title',
  changes: [
    'item 1 ...',
    'item 2 ...',
  ] },
```

约束：

- 永远把新版本插入数组**最前面**（最新在最上）。
- `changes` 用动词起始的英文短句或简明中文短句。
- 若有破坏性变更，请在条目前加 `BREAKING:`。
- `version.json::changes` 与 `App.vue::changelog` 的内容应基本一致（前者面向 OTA 弹窗，后者面向用户 Changelog 弹窗），保持手动同步。
